h1. Aurum: текущее состояние сервиса, сценарии и памятка для сопровождения

_Дата актуализации: 24.04.2026_

h2. 1. Что умеет сервис сейчас

Aurum — внутренний сервис AGIMA для работы с заявками на бюджет и затраты.

Текущий функционал:

* создание заявок и черновиков;
* согласование по ролям;
* автоматическое подключение HOD-маршрута по цехам;
* HOD-валидация прямых затрат в конкурсных заявках;
* дополнительные согласования после старта маршрута;
* отложенное согласование с передачей коллегам;
* передача в оплату;
* планирование полной и частичной оплаты;
* фиксация фактических платежей;
* единая квота AGIMA с разбиением по цехам и тегам;
* ручное списание из тега без заявки;
* комментарии, `@mention`, viewer access, история изменений, таймлайн, email-лог и вложения.

h2. 2. Роли и текущие полномочия

h3. AD

* создает и редактирует свои заявки;
* отправляет на согласование;
* комментирует;
* выдает viewer access к заявке;
* передает заявку в оплату;
* закрывает и заново открывает заявку.

h3. NBD

* участвует в согласовании;
* может видеть широкий список заявок;
* может быть добавлен как дополнительный согласующий.

h3. AI-BOSS

* участвует в согласовании;
* может быть дополнительным согласующим;
* в текущем пользовательском сценарии отдельный тип заявки AI больше не используется как основной путь.

h3. COO

* участвует в согласовании;
* видит все заявки;
* управляет тегами;
* управляет общей квотой AGIMA, цеховыми квотами и теговыми квотами;
* может назначать тег заявке;
* может работать с оплатой так же, как CFD/BUH.

h3. CFD

* участвует в согласовании;
* видит все заявки;
* работает с оплатой;
* управляет тегами и квотами;
* заполняет/меняет источник финансирования и business category;
* может согласовывать как BUH в оплатном контуре.

h3. BUH

* участвует в согласовании и обязателен в любом маршруте;
* видит все заявки;
* ведет оплатный контур;
* планирует и фиксирует полные и частичные платежи;
* меняет сумму оплаты, курс валюты, данные финплана, теги и квоты;
* валидирует прямые затраты в конкурсных строках.

h3. HOD

* это руководитель цеха;
* работает только в пределах своих `hodDepartments`;
* согласует заявки своего цеха;
* валидирует прямые затраты по внутренним специалистам;
* может управлять тегами и квотами своего цеха;
* может вручную списывать сумму в поле `Потрачено без заявок` на уровне тега.

h3. ADMIN

* управляет ролями;
* видит широкий контур заявок;
* может выполнять аварийные и административные действия;
* помогает сопровождать нестандартные сценарии.

h2. 3. Цеха и типы заявок

h3. Цеха

Сейчас в сервисе используются цеха:

* Аккаунтинг
* Транзит
* AI RnD
* Администрация
* Разработка
* Маркетинг
* HR
* Проектирование и дизайн
* Продуктовая аналитика
* Производственный менеджмент
* Outstaff
* Финансово-юридический отдел

h3. Типы заявок по цехам

Для `Аккаунтинг` доступны:

* Welcome-бонус
* Закупка
* Подарки
* Неформальное мероприятие
* Совместный мерч
* Конкурсное задание

Для `Транзит` доступен:

* Транзит

Для остальных цехов основной текущий тип:

* Закупка

Важно:
* старые названия вроде `Закупки сервисов`, `Сервисы/транзиты для клиентов`, `Транзиты для проектов`, `AI-инструмент/подписка` в коде нормализуются как legacy и нужны только для совместимости со старыми данными;
* текущий пользовательский путь должен опираться на `Закупка` и `Транзит`.

h2. 4. Источники финансирования

Актуальные источники:

* `Квоты AGIMA`
* `Отгрузки проекта`
* `Я не знаю`

Правила:

* `Квоты AGIMA` — заявка участвует в единой квотной модели и должна получить тег;
* `Отгрузки проекта` — заявка не должна списываться из квот;
* `Я не знаю` — временное состояние, которое потом уточняет BUH или CFD.

Особый случай:

* если тег заявки равен `Транзит`, квота не списывается даже при finance-классификации.

h2. 5. Единая квотная модель

Сейчас основной квотный сценарий в продукте такой:

# Есть общая квота AGIMA на месяц.
# Она распределяется по цехам.
# Внутри цеха квота раскладывается по тегам.
# Списание идет:
** по заявкам;
** вручную через `Потрачено без заявок`.
# Все изменения квот и ручных списаний попадают в `Историю изменений`.

Кто может редактировать:

* `CFD`
* `COO`
* `BUH`
* `ADMIN`
* `HOD` — только свой цех

Что важно для саппорта:

* основная квотная таблица сейчас — `administrationQuotas`;
* история квот — `quotaChangeLogs`;
* старые `presalesQuotas`, `aiToolQuotas`, `cfdQuotas`, `cooQuotas` остаются в схеме, но не являются главным пользовательским контуром.

h2. 6. Основной маршрут заявки

h3. Создание

# Автор создает заявку.
# Указывает цех, тип заявки, источник финансирования и обязательные поля.
# Может сохранить черновик или отправить на согласование.

h3. Автоматический маршрут согласования

В текущей реализации effective-маршрут складывается из нескольких источников:

* выбранные автором роли;
* обязательный `BUH`;
* `HOD` по выбранному цеху;
* `HOD` по цеху `Финансово-юридический отдел`;
* для конкурсного задания — `HOD` по цехам внутренних специалистов, если там нужна валидация.

Это важно:

* `HOD` сейчас фактически участвует почти во всех живых сценариях, потому что selected department и `Финансово-юридический отдел` автоматически попадают в effective HOD departments;
* если в конкурсной строке стоит `Валидация не требуется`, этот конкретный специалист не создает задачу на валидацию.

h3. Дополнительные согласования

Во время согласования текущий согласующий может:

* просто согласовать;
* отклонить;
* согласовать и отправить дальше;
* отложить согласование и отправить дальше.

Дополнительные роли можно выбрать из:

* `NBD`
* `AI-BOSS`
* `COO`
* `CFD`
* `HOD`

Если выбрали `HOD`, нужно указать конкретный цех.

h2. 7. Конкурсное задание

Текущая модель конкурсного задания:

* есть блок `Внутренние специалисты`;
* есть блок `Подрядчики`;
* общая сумма собирается из прямых затрат;
* для внутренних специалистов может требоваться HOD- или BUH-валидация;
* строка считается провалидированной, если ее подтвердил либо `HOD`, либо `BUH`;
* при необходимости автор может поставить `Валидация не требуется`.

Что смотреть при проблеме:

* `specialists` в `requests`;
* флаги `hodConfirmed`, `buhConfirmed`, `validationSkipped`;
* `requiredHodDepartments`;
* approvals по `HOD`.

h2. 8. Поля и текущие валидации

h3. Базовые поля

Обязательные почти всегда:

* цех;
* тип заявки;
* источник финансирования;
* название;
* обоснование;
* дедлайн согласования;
* `Ожидание затраты`.

h3. Суммы и НДС

* в форме есть суммы без НДС и с НДС;
* одна может рассчитываться автоматически из другой;
* ставка НДС редактируемая;
* если НДС пустой, считается `0%`.

h3. Предоплата

Если включена галочка `Требуется предоплата`, становятся обязательными:

* сумма предоплаты;
* сумма предоплаты с НДС;
* дата предоплаты.

h3. Отгрузки проекта

Если источник финансирования — `Отгрузки проекта`, обязательны:

* `ID и название отгрузки в финплане`;
* `Сумма отгрузки`;
* корректные входящие суммы;
* сопутствующие поля по дате/платежу проекта.

h3. Договор и due diligence

Для заявок больше `100 000` без НДС:

* нужен договор;
* достаточно либо ссылки на договор, либо contract attachment.

Для заявок больше `500 000` без НДС:

* нужна галочка `Проведена должная осмотрительность`;
* нужна ссылка на задачу в Jira.

Исключения:

* `Welcome-бонус`
* `Конкурсное задание`

h3. Ограничения по датам

* `Дедлайн согласования` должен быть не раньше завтрашнего дня;
* `Дедлайн согласования` не может быть позже `Ожидание затраты`;
* `Ожидание затраты` не может быть раньше даты создания заявки;
* дата `Когда платят нам` не может быть раньше `2006`, иначе ошибка `AGIMA тогда еще не было`.

h2. 9. Оплата

Оплатный контур сейчас ведут `BUH` и `CFD`.

Они могут:

* перевести заявку в `Требуется оплата`;
* запланировать полную оплату;
* запланировать частичную оплату;
* добавлять несколько плановых платежей;
* фиксировать фактические оплаты в той же ленте;
* отменять план или платеж;
* отправлять напоминание об оплате.

Текущие статусы оплаты:

* `awaiting_payment`
* `payment_planned`
* `partially_paid`
* `paid`

Для BUH/CFD есть специальные признаки:

* `Есть нераспределенный платеж`
* `Есть остаток к оплате`
* `Оплата запланирована`

h2. 10. Комментарии, viewer access и @mentions

h3. Комментарии

В карточке заявки поддерживаются комментарии и ответы.

h3. @mentions

Если в комментарии упомянуть сотрудника через `@`:

* упоминать можно только `@agima.ru`;
* человеку уходит письмо;
* ему автоматически выдается доступ на просмотр и комментирование заявки.

h3. Viewer access

Автор заявки или `ADMIN` может вручную выдать доступ к заявке:

* это дает просмотр и комментарии;
* не дает права редактировать заявку и согласовывать ее;
* факт выдачи доступа фиксируется в таймлайне.

h2. 11. Теги и business category

h3. Теги

Теги используются для:

* квот;
* фильтров;
* финансовой классификации заявок.

Кто может управлять тегами:

* `CFD`
* `BUH`
* `COO`
* `ADMIN`
* `HOD` — только в своих цехах

h3. Business category

Это отдельное поле классификации, которое не равно `Типу заявки`.

Его используют для фильтрации и сортировки заявок.
Заполнять и менять его могут:

* `BUH`
* `CFD`
* `ADMIN`

h2. 12. Файлы и договоры

Текущие ограничения по вложениям:

* до `20` файлов на заявку;
* до `40 МБ` на файл.

Поддерживаемые форматы:

* PDF
* Word
* Excel
* CSV
* PowerPoint
* изображения
* архивы `zip`, `7z`, `rar`

Есть два типа файлов:

* обычные вложения;
* вложения типа `contract`.

h2. 13. Видимость заявок

Широкий доступ к списку заявок сейчас есть у:

* `NBD`
* `AI-BOSS`
* `COO`
* `CFD`
* `BUH`
* `ADMIN`

`HOD` видит не абсолютно все заявки, а все доступные ему:

* по своим цехам;
* по историческому участию;
* по выданному viewer access.

Пользователь с viewer access видит заявку во `Все заявки`.

h2. 14. Основные статусы заявки

Базовые статусы:

* `draft`
* `hod_pending`
* `pending`
* `approved`
* `rejected`
* `awaiting_payment`
* `payment_planned`
* `partially_paid`
* `paid`
* `closed`

Дополнительные состояния:

* `isCanceled`
* `archivedAt`

h2. 15. Что проверять при инцидентах

h3. Не видно заявку

Проверить:

# Кто автор.
# Какие роли у пользователя.
# Есть ли `viewerAccess`.
# Было ли историческое участие в `approvals`.
# Не архивная ли заявка и не отфильтрована ли датами.

h3. Заявка зависла в согласовании

Проверить:

# `approvals`
# pending approvals
# `requiredRoles`
# `requiredHodDepartments`
# конкурсные строки и их валидацию
# не было ли дополнительного согласования или отложенного решения

h3. Проблема с оплатой

Проверить:

# `status`
# `plannedPaymentAmount` / `plannedPaymentAmountWithVat`
# `plannedPaymentSplits`
# `paymentSplits`
# `paymentResidualAmount`
# `neededBy`
# `paymentPlannedAt`

h3. Проблема с квотой

Проверить:

# источник финансирования
# `cfdTag`
# не равен ли тег `Транзит`
# `administrationQuotas`
# `quotaChangeLogs`
# нет ли ручного списания в `manualSpent`

h3. Не пришло письмо

Проверить:

# `requestEmailLogs`
# `requestTimelineEvents`
# email получателя
# SMTP / `/api/email/send`
# auth/email env
