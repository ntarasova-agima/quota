_Дата фиксации: 27.03.2026_

h2. 1. ЧТО ЭТО ЗА СЕРВИС

Aurum — внутренний сервис AGIMA для согласования затрат, инвестиций и оплат.
Сервис покрывает путь от создания заявки до ее согласования, передачи в оплату, фиксации факта оплаты и закрытия.

*Ключевые сущности:*
* заявка;
* согласование по ролям;
* квоты;
* оплата;
* комментарии;
* файлы;
* история изменений;
* таймлайн событий и писем.

*Сервис уже поддерживает:*
* несколько типов заявок;
* разные маршруты согласования в зависимости от источника финансирования;
* квоты NBD / COO / CFD;
* HOD-валидацию по конкурсным заданиям;
* оплатный контур BUH;
* частичную оплату;
* архивирование старых заявок;
* логирование писем и действий.

h2. 4. РОЛИ В СЕРВИСЕ

AD
* инициатор заявки
* создает, редактирует, отправляет, закрывает свою заявку

NBD
* согласующий по квоте пресейлов
* видит свои заявки и все заявки с источником "Квота на пресейлы"
* ведет свою таблицу квот

COO
* согласующий / держатель части внутренних квот
* работает со своей таблицей квот COO

CFD
* согласующий
* работает с тегами
* работает со своей таблицей квот CFD

BUH
* видит все заявки
* действия выполняет в оплатном контуре
* планирует оплату, ставит оплату, частичную оплату, курс валюты, ID Финплана
* может ставить/создавать теги

HOD
* руководитель цеха / подцеха
* работает только со своими цехами
* валидирует часы и прямые затраты по специалистам в конкурсных заданиях
* конкурсные заявки со специалистами сначала идут к HOD, а только потом к обычным согласующим

ADMIN
* администрирует роли и пользователей
* может выполнять аварийные и операционные действия
* может сопровождать активные заявки архивированных сотрудников

h2. 6. ОСНОВНЫЕ ПОЛЬЗОВАТЕЛЬСКИЕ СЦЕНАРИИ

h3. 6.1. Автор (AD или любой авторизованный сотрудник) создает обычную заявку

# Создает заявку.
# Заполняет тип, источник, поля формы.
# Отправляет на согласование.
# Сервис определяет маршрут согласования.
# Согласующим уходят письма.
# После полного согласования автор:
** либо передает в оплату;
** либо принимает без оплаты, если оплата по счету не требуется.
# После оплаты автор закрывает заявку.

h3. 6.2. Конкурсное задание без специалистов

# Автор создает заявку.
# Указывает данные заявки и согласующих.
# Отправляет как обычную заявку.
# Она сразу уходит в обычное согласование.

h3. 6.3. Конкурсное задание со специалистами

# Автор добавляет специалистов.
# Для каждого специалиста указывает:
** имя;
** цех;
** часы;
** прямые затраты (если знает).
# Поле общей суммы не заполняется вручную.
# Сумма считается автоматически как сумма прямых затрат специалистов.
# Если прямые затраты еще не известны, заявка может быть отправлена без суммы.
# Такая заявка получает статус HOD-этапа и уходит не обычным согласующим, а HOD.
# Каждый HOD получает письмо только по своим специалистам и своим цехам.
# HOD валидирует часы и прямые затраты только в рамках своих цехов.
# После того как все нужные цеха провалидировали данные:
** сумма фиксируется;
** заявка уходит в обычное согласование;
** только тогда обычным согласующим уходят письма.

h3. 6.4. HOD валидирует конкурсную заявку

# HOD открывает раздел "На согласовании".
# Видит заявки, где нужны действия именно по его цехам.
# Открывает заявку.
# Меняет часы / прямые затраты только для специалистов своего цеха.
# Ставит галочку "Подтверждено Цехом".
# Когда подтверждены все нужные цеха, заявка уходит дальше обычным согласующим.

h3. 6.5. Согласующий принимает решение

# Открывает раздел "На согласовании".
# Видит только заявки, требующие его решения.
# Согласует или отклоняет.
# При отклонении комментарий обязателен.
# Если это последняя роль — заявка переходит в "Согласовано".

h3. 6.6. Автор передает согласованную заявку в оплату

# Открывает согласованную заявку.
# Нажимает "Передать в оплату".
# Заявка переходит в статус "Требуется оплата".
# BUH получает письмо.

h3. 6.7. BUH работает с оплатой

# Открывает раздел "На согласовании".
# Видит заявки в статусах:
** Требуется оплата
** Запланирована оплата
** Частично оплачено
# В карточке заявки указывает:
** ID затрат в Финплане
** дату оплаты
** суммы без НДС / с НДС
** курс валюты, если нужно
** тег
# Может:
** запланировать оплату;
** отметить частичную оплату;
** отметить оплату.

h3. 6.8. Частичная оплата

# BUH указывает сумму текущего транша.
# Указывает остаток.
# Указывает следующую дату оплаты.
# Сервис фиксирует транш.
# Допускается до 5 траншей.

h3. 6.9. Автор закрывает заявку

# Если оплата была — закрывает после факта оплаты.
# Если оплата не требуется — использует "Принять без оплаты".
# Закрытую заявку можно снова открыть кнопкой "Открыть заново".
# Она вернется в предыдущий статус:
** Согласовано
** либо Оплачено

h2. 7. СТАТУСЫ ЗАЯВКИ

*Основные статусы:*
* draft — черновик
* hod_pending — ждет валидации цеха
* pending — ожидает согласования
* approved — согласовано
* rejected — не согласовано
* awaiting_payment — требуется оплата
* payment_planned — запланирована оплата
* partially_paid — частично оплачено
* paid — оплачено
* closed — заявка закрыта

*Дополнительные состояния:*
* isCanceled — отменена
* archivedAt — архивная заявка

*Особые правила:*
* закрытая заявка может быть открыта заново;
* для конкурсного задания со специалистами этап HOD обязателен до обычного согласования;
* повторное согласование может запускаться после критичных изменений.

h2. 8. ЛОГИКА СОГЛАСОВАНИЯ И РЕДАКТИРОВАНИЯ

*При критичных изменениях заявка может:*
* остаться в текущем маршруте;
* уведомить уже участвовавших согласующих;
* частично пересобрать маршрут;
* уйти на повторное согласование.

*Ключевые правила:*
* изменение источника финансирования может сменить держателя квоты;
* изменение месяца может потребовать повторного согласования держателя квоты;
* изменение контрагента не сбрасывает согласование, но уведомляет BUH;
* изменение конкурсной заявки со специалистами может снова увести заявку в HOD-этап.

*Все изменения пишутся в:*
* историю изменений;
* таймлайн заявки.

Одно сохранение = одна группа изменений.

h2. 9. КВОТЫ

*В сервисе есть квоты:*
* NBD / пресейлы
* COO / внутренние затраты
* CFD / отдельная квотная логика

*Общие правила:*
* квоты считаются без НДС;
* отрицательный остаток допустим;
* заявки могут быть согласованы даже при уходе квоты в минус.

*Логика списания:*
* после согласования ориентируемся на месяц согласования;
* если BUH меняет дату и сумму оплаты — ориентируемся на последние данные BUH;
* после оплаты финальным считается фактическая дата и сумма оплаты;
* если заявка закрыта без оплаты, квота все равно списывается;
* если фактическая сумма меньше, квота уменьшается до факта.

* письма и ошибки писем логируются в Convex и показываются в таймлайне заявки.

h2. 13. АРХИВАЦИЯ

*Архивация заявок:*
* заявки старше года помечаются архивными;
* по умолчанию не мешают в обычных списках;
* доступны по диапазону дат и при целевом поиске.

*Архивация сотрудников:*
* админ архивирует аккаунт;
* активные заявки переходят в сопровождение админа;
* в заявке сохраняется пометка, кто был исходным автором.

h2. 14. АДМИНСКИЕ ИНСТРУМЕНТЫ

*Админу доступны:*
* управление ролями
* архивирование сотрудников
* напомнить о согласовании
* согласовать как роль
* сопровождение заявок архивированных сотрудников
* просмотр таймлайна и логов писем

*Важно:*

h2. 15. ЧТО ВАЖНО РАЗРАБОТКЕ И СОПРОВОЖДЕНИЮ

*Критические зоны логики:*
* requests.ts — центральный модуль продукта; любые изменения здесь надо проверять особенно внимательно
* квоты и BUH-логика — самая чувствительная часть сервиса
* HOD-сценарий — важно не пустить обычных согласующих раньше полной валидации цехов
* email-логика — важно не дублировать письма лишним людям

*Что проверять после каждого значимого изменения:*
* npm run build
* npx convex dev --once --tail-logs disable
* сценарии:
** обычная заявка
** конкурсное задание без специалистов
** конкурсное задание со специалистами
** HOD-валидация
** BUH-оплата
** частичная оплата
** повторное согласование после редактирования
** закрытие / открыть заново

h2. 16. ЧТО ВАЖНО ИНФРАСТРУКТУРЕ

* сервис зависит от Convex и SMTP-сервер agima.ru
* нужна стабильная работа frontend и SSL
* домен и env-переменные критичны для писем и авторизации
* при проблемах с почтой смотреть:
** таймлайн заявки
** requestEmailLogs в Convex
* при проблемах с ролями / доступом смотреть:
** таблицу roles
** таблицу requests
** approvals
