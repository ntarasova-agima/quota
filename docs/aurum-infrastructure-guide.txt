AURUM
Памятка для инфраструктуры, эксплуатации и технической поддержки
Дата фиксации: 27.03.2026

==================================================
1. ОБЩЕЕ НАЗНАЧЕНИЕ
==================================================

Aurum — внутренний сервис AGIMA для маршрутизации заявок на затраты, согласования, оплаты, квот и аудита действий.

Сервис завязан на:
- frontend-приложение Next.js;
- Convex как backend и data layer;
- Resend как email-провайдера.

==================================================
2. ТЕКУЩИЙ ТЕХНОЛОГИЧЕСКИЙ СТЕК
==================================================

Frontend:
- Next.js 16
- React
- Tailwind CSS
- shadcn/ui

Backend:
- Convex
- convex-auth

Email:
- Resend

Авторизация:
- email + код
- JWT / JWKS для convex-auth

==================================================
3. ПРОДОВОЕ РАЗМЕЩЕНИЕ
==================================================

На текущем этапе реальный прод-контур предполагает:
- размещение Next.js frontend на VM;
- использование внешнего Convex deployment;
- использование Resend для отправки почты.

Это не полностью self-hosted система на одной VM.

==================================================

==================================================
6. ENV-ПЕРЕМЕННЫЕ
==================================================

Критичные переменные:
- CONVEX_DEPLOYMENT
- NEXT_PUBLIC_CONVEX_URL
- NEXT_PUBLIC_CONVEX_SITE_URL
- SITE_URL
- RESEND_API_KEY
- RESEND_FROM
- JWT_PRIVATE_KEY
- JWKS

Важно:
- ошибки в SITE_URL и auth env ломают сценарии авторизации по коду;
- ошибки в email env ломают все уведомления;
- ошибки в Convex env ломают весь сервис.

==================================================
7. EMAIL И ДОМЕННАЯ ЧАСТЬ
==================================================

Используется Resend.
Текущий домен отправителя:
- aurum.agima.ru

Рекомендуемый отправитель:
- no-reply@aurum.agima.ru

Если пользователи жалуются, что письма не пришли, проверять по порядку:
1. таймлайн заявки в интерфейсе;
2. email log в Convex;
3. наличие и корректность RESEND_API_KEY;
4. доменную настройку у почтового провайдера;
5. SPF / DKIM / DMARC / доменную верификацию.

Аватарка отправителя в почте не управляется из Aurum напрямую.
Для этого обычно нужны доменные и почтовые настройки уровня BIMI / провайдера.

==================================================
8. ЧТО НУЖНО ЗНАТЬ САППОРТУ
==================================================

Основные пользовательские инциденты:
- не пришло письмо;
- не пускает по коду;
- заявка не видна в нужном разделе;
- заявка не ушла нужному согласующему;
- BUH / HOD / NBD не видят свою рабочую очередь;
- квота посчиталась неожиданно;
- заявка не закрывается / не открывается заново.

Первая линия проверки:
1. Открыть карточку заявки.
2. Проверить вкладки:
   - Изменения
   - Таймлайн
3. Проверить статус заявки.
4. Проверить approvals.
5. Проверить роли пользователя.
6. Проверить email-лог по заявке.

9. ЧТО НУЖНО ЗНАТЬ РАЗРАБОТКЕ
==================================================

Главный модуль продукта:

Самые чувствительные места:

- создание конкурсного задания;
- HOD-валидацию;
- переход в оплату;
- частичную оплату;
- оплату;
- закрытие;
- открыть заново;
- повторное согласование.

10. КЛЮЧЕВЫЕ БИЗНЕС-ПРАВИЛА, ВАЖНЫЕ ДЛЯ ПОДДЕРЖКИ
==================================================

1. Не все заявки идут сразу согласующим.
- конкурсные задания со специалистами сначала идут к HOD.

2. Сумма по конкурсному заданию может собраться не сразу.
- если прямые затраты неизвестны, автор может отправить без суммы;
- сумма сформируется после валидации цехов.

3. BUH видит все заявки, но действия делает только в оплатном контуре.

4. NBD видит:
- свои заявки;
- все заявки, которые берутся из квоты на пресейлы.

5. Если сотрудник архивирован:
- активные заявки переходят в сопровождение админа;
- исходный автор в карточке сохраняется как историческая информация.

6. Закрытая заявка не окончательно запечатана.
- ее можно открыть заново;
- она вернется в предыдущий статус.

12. ЭКСПЛУАТАЦИОННЫЕ РИСКИ
==================================================

Самые рискованные зоны:
- requests.ts как единый центр бизнес-логики;
- квоты при смене дат и сумм;
- частичная оплата;
- HOD-сценарий по нескольким цехам;
- несогласованность статуса заявки и approvals;
- ошибки почты, которые внешне выглядят как “сервис завис”.

13. РЕКОМЕНДУЕМЫЕ ОПЕРАЦИОННЫЕ ПРОВЕРКИ
==================================================

После деплоя или крупной правки:
1. npm run build
2. npx convex dev --once --tail-logs disable
3. ручной smoke test:
   - sign-in по коду
   - создание обычной заявки
   - создание конкурсного задания со специалистами
   - HOD-валидация
   - согласование обычной ролью
   - передача в оплату
   - BUH-оплата
   - закрытие заявки
   - открытие заново
